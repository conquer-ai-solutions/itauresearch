Data Ingestion &
Research Architecture

How we solve the Block 1 problem: getting data from Bloomberg, Capital IQ, Economatica, CVM, IBGE, CONAB, ANP, and internal sources into a unified research layer — while respecting every licensing constraint.
The Core Constraint
Why AI Cannot Touch Data Sources Directly
Licensed data providers explicitly block direct AI agent access. This is the non-negotiable architectural constraint that shapes every design decision.
Licensing Reality
Capital IQ (S&P): Explicitly prohibits AI agents, MCP servers, and automated AI-driven systems from connecting to their APIs. The CIQ Pro API exists — but the caller must be a traditional microservice, not an AI agent. Bearer tokens expire every 60 minutes and cannot be issued to AI processes.

Bloomberg: AI/ML use of Bloomberg data requires a specific license amendment to the existing contract. Standard terminal or BLPAPI licences do not permit feeding data into AI systems. Data License (DL) with the AI/ML amendment is the compliant path.

Economatica: Commercial license with distribution restrictions. Programmatic REST API may be available but terms must be reviewed for AI-derived-output use cases.

The implication: Any architecture where an LLM, AI agent, or MCP server connects directly to these systems is non-compliant and puts Itaú at licensing risk. The architecture must enforce a hard boundary between data ingestion and AI consumption.
Solution Architecture
Decoupled Two-Layer Design
Traditional data engineering (Layer 1) handles ingestion and staging. A separate AI layer (Layer 2) reads only from the staged data lake. The red boundary between them is absolute — AI never crosses it.

Freshness SLAs
Public sources: data available in lake within 15 min of publication. Licensed sources: within 1 hour of delivery. Analyst exports: within 5 min of upload. Every record carries a freshness timestamp — AI flags stale data in outputs.
Storage Estimate
Year 1: ~50GB raw + ~20GB normalized (structured data). ~200GB for document embeddings (internal docs). Total: ~270GB. Well within single-instance Postgres/S3 range. No distributed database needed at this scale.

Scenario A — CIQ Pro API Available
Automated via middleware microservice
Standard Python/Java microservice (not AI) authenticates with OAuth bearer tokens. Tokens expire every 60 minutes — auto-refresh via dedicated auth module. REST/HTTPS calls for fundamentals, estimates, financials, M&A, ownership. JSON → normalize → lake. Nightly bulk + on-demand. Rate limits managed via queue with exponential backoff.
CIQ Pro API→ Auth (60min bearer)→ REST Microservice→ JSON → Normalize→ Data Lake
Scenario B — Xpressfeed Bulk Delivery
Automated via SFTP listener
S&P pushes pipe-delimited flat files via SFTP nightly. Critical: files have no column headers — raw pipe-separated values. Parsing requires S&P's schema docs mapping column positions to field names. SFTP listener detects drops, applies schema map, handles type casting (proprietary date formats, null indicators), validates row counts against manifest. Files can be 500MB+ for universe snapshots.
S&P SFTP Push→ SFTP Listener→ Schema Mapper (no headers)→ Type Cast + Validate→ Data Lake
Scenario C — Excel Plugin Only
Analyst-driven export + automated ingest
Analysts use CIQ()/CIQF() Excel formulas. Export CSV/XLSX → upload portal. Ingest service validates, normalizes, cross-checks against known CIQ field IDs. We template the Excel sheets for consistent pulls — reduces variability and speeds QA. More manual, fully compliant.
Analyst → Excel CIQ()→ CSV Export→ Upload Portal→ Validate + Normalize→ Data Lake

The Hard Problem
Private Company Research — Agentic Mosaic
Everything above assumes structured data from known sources with pre-built pipelines. Private companies break that assumption entirely.
Why This Requires a Different Architecture
For public companies, we pre-stage data in the lake. For private companies, we can't — we don't know in advance which company the analyst will research, and no structured source covers them. The solution: an agentic orchestration layer that assembles a data mosaic in real time, with confidence scoring on every data point.
1
Sector Identification
Analyst: "Research Company X in sugar/ethanol." AI identifies sector, geography, relevant dimensions — commodity pricing, agricultural production, regulatory, comparable publics.
2
Public Comparable Pull
Queries lake for CVM filings and fundamentals of public peers — Raízen, São Martinho, Jalles Machado. Already staged. Provides valuation benchmarks.
3
Sector Macro Pull
Pre-staged IBGE sector GDP, CONAB sugarcane forecasts, ANP ethanol pricing. All normalized and ready in the lake.
4
Agentic Web Search
Real-time: company news, Receita Federal (CNPJ), industry associations, IR materials, press releases, government contracts. AI IS the ingestion layer — but only for public/web sources.
5
Internal Knowledge
Semantic search across Salesforce for prior engagements, pitchbooks, CRM notes. Surfaces institutional knowledge about the company or sector.
6
Mosaic Assembly
Full data → 1M-token context. AI builds company profile. Every point tagged: "Revenue R$3.4B per CVM" vs. "est. R$800M–1.2B from sector benchmarks" with confidence scores.
AI Research Engine
How the Engine Consumes the Lake
The AI engine reads exclusively from the data lake via a standard query API. It is completely source-agnostic.
1. Research Brief
Analyst: "Coverage initiation on Cosan" or "Sector thematic on Brazilian ethanol" or "Private company pitch for Company Y."
2. Data Requirements Planning
Engine determines which lake domains, companies, time periods, and metrics. Generates structured query against lake API.
3. Lake Query
Standard REST queries. Receives structured data package — financials, market data, filings, macro, internal docs. All provenance-tagged.
4. 1M-Token Context Assembly
Full data package loaded simultaneously: financial statements + market data + CVM filings + CONAB extracts + IBGE macro + prior internal research. This is where large context matters.
5. Research Synthesis
Generates: investment thesis, financial summary, peer comparison, risk factors, sector outlook. Every claim cited to specific source, period, confidence level.
6. Output → Block 2
Structured research feeds presentation/report generation — already demonstrated. Block 1 solved → Block 2 already works.
Enterprise Considerations
Governance, Security, Compliance
Data Governance & Provenance
Every record tagged: source system, pull date, license type, freshness. AI outputs cite sources. Analyst traces any claim to originating data point. Compliance audit: "did AI access Bloomberg directly?" → No, with full lineage proof.
Security & Data Residency
Lake + AI engine in Itaú's VPC (or AWS Bedrock). No data leaves perimeter. Licensed data stays within existing license terms — processed and stored on their infrastructure.
Freshness & Staleness
Configurable refresh per source. Bloomberg/CIQ: daily+. CVM: on filing. IBGE: publication cadence. CONAB: monthly. Staleness indicators so AI flags outdated data in outputs.
Extensibility
New source = one new ETL adapter. Lake schema extensible. AI unchanged. Future sources (BCB, B3 direct, alternative data) plug in without rearchitecting.
